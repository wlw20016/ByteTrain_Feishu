产品先跑起来，再把工程做扎实

项目演进分成两个阶段：先用 Android + Gradle 快速验证产品形态和 UI 链路，再用 Bazel、OpenSpec、VSCode 插件和 AI 诊断上下文把项目推进到可管理、可验证、可诊断。

Android + Gradle

快速成型阶段

完成 Android App 入口、Messages 和 Mail 底部 Tab、列表与详情页面，以及 10000 条 mock/runtime 数据的分页展示。

  • 消息列表、消息详情、邮箱列表、邮箱详情
  • 滚动到底部附近触发分页加载

Android 核心实现：复用展示，不混淆业务

消息和邮箱在领域模型上保持独立，但 UI 层统一映射为 `UnifiedListItem`。这样既保留业务语义，又减少列表和详情页面的重复展示逻辑。

Unified UI Model

业务模型映射为统一展示模型

MessageItemMessageUiMapper → UnifiedListItem → 消息列表 / 详情
MailItemMailUiMapper → UnifiedListItem → 邮箱列表 / 详情
displayStyle = DENSE_CONVERSATION
displayStyle = MAIL_CARD
badges + detail 共享展示结构
Pagination

10000 条数据分页与性能优化

采用 cursor 分页 + 屏幕可见数量分页。分页大小不是写死，而是根据列表区域高度和预估 item 高度动态计算，并增加少量预加载缓冲。

Messages：按 `64dp` 预估消息行高计算可见行数
Mail：按 `92dp` 预估邮箱卡片高计算可见卡片数
page size = 可见数量 + 2 条预加载，最终限制在 `1..200`
